## 一、功能概述

实现了"从产物创建开发任务"功能：用户可以从需求分析流程产出的 User Story 产物中，选择 stories 并自动启动开发流程。

支持两种流程模板：
- **标准流程**（`simple-dev-from-artifact`）：加载产物 → 生成变更名称 → 生成任务清单 → 实施代码 → Agent 代码审核 → 最终审核
- **完整流程**（`openspec-dev-from-artifact`）：加载产物 → 生成变更名称 → 生成 OpenSpec → 人工审核 Spec → 实施代码 → Agent 代码审核 → 最终审核 → 归档 Spec

## 三、核心技术方案

### 3.1 system_init 节点类型

新增 `system_init` 节点类型，用于在流程启动时将 flow 变量加载为节点输出，供下游节点通过 `{{nodes.load_artifact.outputs.xxx}}` 引用。

**关键设计 — `output_from_variables`**：

直接从 `flowRun.Variables` 读取原始变量值，绕过 YAML 模板渲染。解决了 `RenderParams` 将含 `---` 的 markdown 内容替换到 DSL 字符串中导致 YAML 解析失败的问题。

```yaml
config:
  output_from_variables:
    - key: user_stories
      variable: selected_stories
```

### 3.2 POST /api/tasks/from-artifact

创建任务 + flow_run + 启动流程的一体化端点：
1. 根据 flowType 查找对应的 workflow template
2. 如果该项目没有对应的 workflow，自动从 template 创建
3. 将选中的 stories 拼接为 markdown 存入 flow variables
4. 调用 orchestrator.startFlow 启动流程

### 3.3 前端对话框

`CreateTaskFromArtifactDialog` 组件功能：
- 按 projectId 加载 user_story 类型产物
- 解析 markdown 中的 `#### US-XXX: title (P0, 5SP)` 格式
- Story 勾选 + SP 合计
- 流程类型单选（标准/完整）
- 自动生成任务标题

## 四、调试过程中发现和修复的问题

### 4.1 YAML 解析失败（已修复）

**问题**：`RenderParams` 在 `StartFlow` 中将 `selected_stories` 内容（含 `---` markdown 分隔符）替换到 DSL 字符串中，YAML 解析器将 `---` 当成文档分隔符。

**修复**：引入 `output_from_variables` 配置，system_init 节点在运行时直接从 `flowRun.Variables` 读取值，完全绕过 YAML 模板替换。

### 4.2 validateRenderedPrompt 误报（已修复）

**问题**：校验逻辑用正则 `^.{2,20}[:：]\s*$` 检测"空字段"，但 prompt 中的段落标题（如 `实施步骤：`、`验收标准：`）触发误判，导致 `implement_code` 节点无法执行。

**修复**：改为只检查 prompt 中是否残留未解析的 `{{ }}` 模板变量。

### 4.3 validateAgentOutput 对 execute/opsx_apply 过严（已修复）

**问题**：agent 完成了代码实施（git 提交成功）但 stdout 没有输出标准 result 字段（容器内 Python 运行时缺失导致 agent 异常退出），校验判定 output 为空。

**修复**：`execute` 和 `opsx_apply` 模式的核心产出是 git 提交，跳过输出长度校验。

### 4.4 手动标记 completed 后不推进 DAG（已修复）

**问题**：手动在数据库中标记 node_run 为 completed 后，重启 orchestrator 不会自动推进后续节点。

**修复**：在 `executor.Start` 恢复阶段新增步骤，对所有 running 状态的 flow_run 调用 `advanceDAG`。

### 4.5 simple 流程 final_review 缺 artifact_scope（已修复）

**修复**：添加 `artifact_scope: predecessor`，与正常流程保持一致。

### 4.6 openspec 流程 code_review timeout 缩进错误（已修复）

**问题**：`timeout: 10m` 写在了 node 级别而非 config 内部。

**修复**：移到 config 下。

### 4.7 GitHub PAT 缺 workflow scope（非代码问题）

**问题**：agent 生成了 `.github/workflows/ci.yml`，但 PAT 没有 `workflow` scope，push 被 GitHub 拒绝。

**解决**：在 GitHub Settings 中给 PAT 添加 `workflow` scope。

## 五、已知限制和待改进项

| 项目 | 说明 |
|------|------|
| 前端 DSL 校验白名单 | `system_init` 不在前端 `VALID_NODE_TYPES` 中，工作流编辑页会显示校验警告，不影响运行 |
| Agent 容器运行时 | agent Docker 镜像只有 Node.js，如果目标项目需要 Python 等其他运行时，agent 无法执行测试验证 |
| Agent 提交 node_modules | agent 实施代码时可能将 `node_modules`/`.pnpm-store` 提交到 git，需要在 prompt 或容器初始化中加 `.gitignore` |
| Retry 粒度 | 当前 retry 会重新执行整个 agent 节点（重跑代码实施），无法只重试 push 等后置步骤 |
| 敏感信息脱敏 | 诊断查询可能将 API key 等敏感配置暴露到 LLM 对话上下文中，建议通过 Droid PostToolUse hook 实现 regex 脱敏 |

## 六、运维操作参考

### Seed 模板

```bash
pnpm --filter @workgear/api run db:seed:templates
```

### 手动推送 bare repo 中的分支

```bash
cd /Users/simon/code/workgear/.cache/repos/project-<PROJECT_ID>/bare.git
git push origin "refs/flows/<FLOW_RUN_ID>/integration:refs/heads/feat/<BRANCH_NAME>" --force
```

### 手动推进卡住的 DAG

如果 node 被手动标记 completed 但后续节点没推进：
1. 重启 orchestrator（会自动 re-advance）
2. 或手动将下一个节点的 status 从 `pending` 改为 `queued`

### Agent 配置热重载

在 UI 修改 agent 角色的 provider/model 后，保存会自动触发 `ReloadAgentConfig` gRPC 调用，不需要重启 orchestrator。注意角色级 `provider_id` 优先级高于全局默认 provider。
