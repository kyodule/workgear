# Proposal: 移动端响应式适配

## 背景（Why）

### 用户痛点

当前系统主要针对桌面端设计，在移动设备（手机、平板）上访问时存在以下问题：

1. **布局不适配移动屏幕**：看板列、任务卡片、对话框等组件在小屏幕上显示不完整，需要频繁横向滚动才能查看内容。

2. **交互元素过小**：按钮、链接、表单输入框等交互元素在触摸屏上难以精确点击，容易误触。

3. **导航体验差**：侧边栏导航在移动端占据过多空间，且无法自动收起，影响内容区域的可用空间。

4. **对话框和弹窗不适配**：DraggableResizableDialog、Markdown 预览等对话框在小屏幕上可能超出视口，无法完整查看或操作。

5. **表格和列表难以浏览**：任务列表、时间线事件、Flow 节点等在移动端横向空间不足，信息密度过高。

### 根因分析

1. **缺少响应式断点设计**：组件未使用 Tailwind 的响应式类（sm:、md:、lg:），布局固定为桌面端尺寸。

2. **固定宽度和高度**：部分组件使用固定像素值（如 `w-[800px]`），在小屏幕上无法自适应。

3. **触摸交互未优化**：按钮和链接的最小点击区域未达到移动端推荐的 44x44px 标准。

4. **导航栏未响应式处理**：侧边栏在移动端未自动折叠为汉堡菜单或底部导航。

5. **对话框未适配移动端**：对话框在移动端未切换为全屏或底部抽屉模式。

## 目标（What）

为系统增加移动端响应式支持，确保在手机和平板设备上提供良好的用户体验。

### 具体方案

1. **响应式布局适配**：
   - 看板列在移动端切换为单列垂直布局或横向滑动
   - 任务详情页在移动端使用全屏模式，标签页切换为底部导航
   - 侧边栏在移动端自动折叠，通过汉堡菜单或底部导航访问

2. **对话框移动端优化**：
   - DraggableResizableDialog 在移动端切换为全屏模式
   - Markdown 预览对话框在移动端占据全屏
   - 表单对话框（创建任务、编辑工作流等）在移动端使用底部抽屉或全屏模式

3. **触摸交互优化**：
   - 按钮和链接的最小点击区域增加到 44x44px
   - 表单输入框高度增加，便于触摸输入
   - 拖拽功能在移动端禁用或替换为长按触发

4. **内容密度调整**：
   - 任务卡片在移动端增加内边距，减少信息密度
   - 时间线事件和 Flow 节点在移动端使用更大的字体和间距
   - 表格在移动端切换为卡片式布局

5. **导航优化**：
   - 顶部导航栏在移动端保持固定，显示项目名称和汉堡菜单
   - 侧边栏导航在移动端切换为抽屉式菜单
   - 任务详情页的标签页在移动端切换为底部固定导航

### 用户体验改进

| 场景 | 改进前 | 改进后 |
|------|--------|--------|
| 浏览看板 | 需要横向滚动查看所有列 | 单列垂直布局或横向滑动切换列 |
| 查看任务详情 | 对话框超出屏幕，无法完整查看 | 全屏模式，标签页切换为底部导航 |
| 点击按钮 | 按钮过小，容易误触 | 按钮增大到 44x44px，易于点击 |
| 导航切换 | 侧边栏占据过多空间 | 侧边栏折叠为汉堡菜单 |
| 查看时间线 | 事件内容被截断 | 内容自适应屏幕宽度，字体和间距增大 |

### 不涉及

- 后端 API 无变更（响应式适配仅涉及前端）
- 数据库 schema 无变更
- 业务逻辑无变更（仅优化展示和交互）

## 非目标

- 不开发独立的移动端 App（仅优化 Web 端响应式）
- 不重构现有组件架构（仅增加响应式样式）
- 不改变桌面端的现有布局和交互（保持向后兼容）
- 不支持横屏模式的特殊优化（优先适配竖屏）

## 风险评估

- **风险等级：中** — 涉及多个核心页面和组件的样式调整，需要充分测试
- 响应式适配可能影响桌面端的现有布局，需要确保向后兼容
- 触摸交互优化可能与现有的拖拽功能冲突，需要特殊处理
- 移动端性能可能受到影响，需要优化资源加载和渲染
