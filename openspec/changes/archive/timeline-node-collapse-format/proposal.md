# Proposal: 统一 Timeline 和 Flow 节点的折叠展开格式

## 背景（Why）

### 用户痛点

在当前系统中，Timeline Tab 和 Flow Tab 都展示节点信息，但两者的展示格式存在不一致：

1. **Timeline Tab 缺少折叠/展开功能**：Timeline 事件以固定格式展示所有内容，当事件数量较多或内容较长时，页面会变得冗长难以浏览。用户无法快速定位关键事件。

2. **Flow Tab 的节点展示格式更优**：Flow Tab 中的 `NodeRunItem` 组件支持折叠/展开，用户可以点击节点头部来查看详细信息。这种交互模式更符合用户对大量信息的浏览习惯。

3. **格式不一致导致用户困惑**：同一个任务的两个标签页使用不同的展示逻辑，用户在切换标签时需要适应不同的交互模式，降低了使用体验的一致性。

### 根因分析

1. **Timeline Tab 设计初期未考虑内容折叠**：`timeline-tab.tsx` 直接渲染所有事件内容，没有状态管理来控制展开/折叠
2. **Flow Tab 的 NodeRunItem 已有成熟的折叠逻辑**：使用 `expanded` 状态和点击事件来控制内容显示
3. **两个组件独立开发**：Timeline 和 Flow 功能在不同时期开发，未统一交互模式

## 目标（What）

为 Timeline Tab 增加节点折叠/展开功能，使其与 Flow Tab 的交互模式保持一致。

### 具体方案

1. **Timeline 事件项增加折叠/展开状态**：
   - 为每个 Timeline 事件项增加 `expanded` 状态（默认折叠）
   - 点击事件头部可切换展开/折叠状态
   - 折叠时仅显示事件类型、时间戳和简短摘要
   - 展开时显示完整的事件内容

2. **统一节点展示格式**：
   - Timeline 事件头部格式与 Flow 节点头部保持一致（图标 + 标题 + 状态标签）
   - 使用相同的视觉样式（边框、间距、hover 效果）
   - 折叠/展开动画保持一致

3. **优化长内容显示**：
   - 折叠状态下，事件内容截断为 1-2 行摘要
   - 展开状态下，显示完整内容（支持 JSON 格式化、代码高亮等）
   - 对于 `agent_dispatch_completed` 等结构化事件，保持现有的格式化展示

### 用户体验改进

| 场景 | 改进前 | 改进后 |
|------|--------|--------|
| 浏览大量事件 | 所有内容全部展开，页面冗长 | 默认折叠，快速浏览事件类型和时间 |
| 查看事件详情 | 直接显示，无需交互 | 点击展开查看详细内容 |
| 切换 Tab | Timeline 和 Flow 交互模式不一致 | 两个 Tab 使用统一的折叠/展开交互 |

## 影响范围（Scope）

### 涉及模块

| 模块 | 影响 | 说明 |
|------|------|------|
| flow-engine | Spec 更新 | Timeline 事件展示规范更新 |
| web (timeline-tab) | 代码变更 | 增加折叠/展开状态管理和交互逻辑 |

### 涉及文件

| 文件路径 | 变更类型 | 说明 |
|----------|----------|------|
| `packages/web/src/pages/kanban/task-detail/timeline-tab.tsx` | MODIFY | 增加 TimelineEventItem 组件，实现折叠/展开逻辑 |
| `packages/web/src/pages/kanban/task-detail/__tests__/timeline-tab.test.tsx` | ADD | 新增单元测试（如果不存在） |

### 不涉及

- 后端 API 无变更（Timeline 数据结构不变）
- 数据库 schema 无变更
- Flow Tab 无变更（保持现有行为）
- TimelineEvent 类型定义无变更

## 非目标

- 不重构 Timeline 的数据加载逻辑（仅优化展示）
- 不改变 Timeline 事件的内容格式（保持现有的 JSON 展示、Badge 样式等）
- 不引入虚拟滚动或分页（当前事件数量不需要）
- 不修改 Flow Tab 的 NodeRunItem 组件（已经符合预期）

## 风险评估

- **风险等级：低** — 纯前端展示逻辑优化，不涉及后端和数据库
- Timeline Tab 是独立组件，变更不影响其他功能
- 折叠/展开是纯 UI 交互，不改变数据流
- 可以通过默认展开状态来保持向后兼容（如果需要）
